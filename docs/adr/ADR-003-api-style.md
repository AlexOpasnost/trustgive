## Context

TrustGive frontend is a separate React + Vite + TailwindCSS v4 app (DESIGN.md Â§11), shipped on Cloudflare Pages, talking to the Django backend over HTTP. The backend additionally serves an RSS feed and an OpenAPI-driven Swagger UI. We need to lock the API paradigm before any code is written so the Frontend Developer can begin building components against a typed contract in Phase 4.

Constraints:
- $0/mo budget â€” no hosted GraphQL gateway
- One solo dev â€” minimise unfamiliar tooling
- SEO-critical (server-rendered pages) â€” must cache cleanly at CDN edge
- TypeScript-first frontend â€” needs typed API client without hand-writing
- 8-week MVP timeline

## Decision

Use **REST over JSON** with the **OpenAPI 3.1** specification, generated by **`drf-spectacular`** from Django REST Framework code. Frontend consumes a **TypeScript client** generated from the OpenAPI doc via `openapi-typescript` + `openapi-fetch`.

**Conventions** (locked):
- All resource paths under `/api/` namespace
- Plural nouns: `/api/charities/`, `/api/causes/`
- Slugs as URL identifiers (ADR-001), not numeric IDs
- snake_case JSON field names (Django/DRF native)
- Standard HTTP semantics: GET for reads, POST for the single write (`/api/events/donation-redirect/`)
- Pagination on every list endpoint (`PageNumberPagination`, default 20)
- Uniform error envelope (`{error: {code, message, details, request_id}}`)
- Documented at `/api/docs/` (Swagger UI), raw schema at `/api/schema/`

**Versioning**: deferred. MVP ships unversioned (`/api/...`). When breaking changes are needed in v2:
- Promote to header-based version negotiation (`Accept-Version: 1` defaulting to current; `Accept-Version: 2` opts into v2)
- OR URL-prefix (`/api/v2/...`) â€” chosen at v2-design time
- Either path is non-breaking for v1 clients
- `info.version` in OpenAPI doc carries the canonical version string

## Consequences

### Positive
- **Mature ecosystem**: drf-spectacular generates OpenAPI from existing DRF serializers/viewsets â€” minimal annotation overhead.
- **Auto-generated typed client**: Frontend Developer runs one command, gets `paths`, `components`, fully typed `api.GET("/api/charities/")` calls. Zero hand-maintained types.
- **CDN-cacheable**: GET requests with deterministic URL + cache headers cache cleanly at Cloudflare. GraphQL would defeat this (single POST endpoint).
- **OpenAPI tooling**: Postman, Hoppscotch, Stoplight, Redocly all consume the same doc â€” useful for partner integrations later, useful for QA in Phase 4.5.
- **Familiar to solo dev**: Alex doesn't need to learn a new query language, schema-stitching, dataloader-batching.
- **Survives the dev-tools tab**: backend devs and frontend devs can hit endpoints with `curl` directly during debugging.

### Negative
- **Over-fetching for some clients**: e.g. catalog list returns full `CharitySummary` even when only `name + slug` is needed. Mitigation: ship `?fields=` query parameter if real bottlenecks appear (no premature optimisation).
- **N+1 risks** in detail endpoints: must use `select_related` / `prefetch_related` aggressively (per backend-developer rules, monitored in Phase 4.5 by Performance Engineer).
- **Versioning is deferred**: future breaking changes require a coordinated FE/BE release. Mitigated by additive-only schema evolution policy (new fields are non-breaking; rename/remove â†’ version bump).

### Neutral
- drf-spectacular adds ~1.5MB to the Docker image â€” irrelevant on Railway.
- Swagger UI adds an unauthenticated `/api/docs/` endpoint. Acceptable since the API itself is unauthenticated (ADR-002).

## Alternatives considered

| Alternative | Rejected because |
|---|---|
| **GraphQL** (e.g. Strawberry or Graphene) | (a) Worse CDN caching â€” single POST endpoint, response varies per query string; (b) adds a query-language Frontend has to learn; (c) auto-generated TS clients exist but require persisted-query setup for caching; (d) no MVP feature requires GraphQL's flexibility (we have ~10 endpoints, all known-shape); (e) adds dataloader/N+1 protection complexity beyond what we need. |
| **gRPC / Protobuf** | No browser-native client; we'd need grpc-web + envoy proxy. Overkill for a 10-endpoint public API. |
| **tRPC** | Couples Frontend and Backend into one TypeScript codebase. We deliberately want clean separation (Django + React, two repos in spirit even if mono-repo'd) so the contract is explicit and testable. |
| **JSON-RPC** | Loses HTTP semantics (every request is POST), defeats CDN caching, no widely-adopted spec format. |
| **HAL / JSON:API hypermedia** | Adds spec ceremony (`_links`, `_embedded`, `relationships`) for no consumer benefit â€” we have one frontend, no third-party integrators. |
| **Plain JSON without OpenAPI** | Loses typed client generation, loses auto Swagger docs, loses contract testing. drf-spectacular is ~1 day of setup for these benefits. |

## Implementation hooks (for Phase 3)

- `pip install drf-spectacular==0.27.2`
- `SPECTACULAR_SETTINGS = {'TITLE': 'TrustGive API', 'VERSION': '1.0.0', 'SERVE_INCLUDE_SCHEMA': False, 'COMPONENT_SPLIT_REQUEST': True}`
- Per-view: `@extend_schema(summary=..., description=..., tags=[...], responses={200: SerializerName, 404: ...})`
- CI: `python manage.py spectacular --validate --fail-on-warn` to catch drift between code and contract
- Frontend: `npx openapi-typescript http://localhost:8000/api/schema/ -o frontend/web/src/api/schema.d.ts` in `package.json` `scripts.gen-api`
