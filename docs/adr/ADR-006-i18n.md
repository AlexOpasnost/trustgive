# ADR-006 — i18n strategy: nested LocalizedString objects + manual RU curation

- **Status**: Accepted
- **Date**: 2026-05-05
- **Deciders**: Backend Developer, per DESIGN.md §13 (handoff checklist) + SPEC.md §3 row 6
- **Supersedes**: —

## Context

TrustGive ships **English + Russian at launch** — a primary differentiator (MARKET_ANALYSIS §1 + §8). Both languages are first-class, not auto-translated. The frontend uses `i18next` for UI string localisation, but **content data** (charity names, descriptions, money-breakdown line labels, methodology notes, source-document labels) must come from the API in both languages.

Per DESIGN.md §13: *"All localised fields ship both `_en` and `_ru` variants OR a `i18n: { en, ru }` object — designer prefers nested."*

Data sources are mostly English-only:
- ProPublica: English only
- Every.org: English only
- CharityBase: English only
- Минюст: Russian only (the 30 manually curated RU records)

We need a strategy that:
1. Ships clean `{en, ru}` payloads to the frontend
2. Allows manual RU curation overlay on English-sourced charity records
3. Survives adding a third language in v2 (e.g. Spanish for LATAM expansion) without table migrations
4. Doesn't bloat every model with `_en`/`_ru` paired columns

## Decision

### Storage: per-field JSONB column with a Django field abstraction

Every localised field is stored as a **JSONB object** keyed by ISO 639-1 language code:

```python
# apps/core/fields.py
class LocalizedTextField(models.JSONField):
    """JSONB column storing {en: str, ru: str, ...}; serialised as nested object."""
    description = "Bilingual text stored as JSONB {en, ru}"
    def from_db_value(self, value, expression, connection):
        if value is None: return {'en': '', 'ru': ''}
        return value
    def get_prep_value(self, value):
        if value is None: return {'en': '', 'ru': ''}
        return value

# apps/charities/models.py
class Charity(models.Model):
    name        = LocalizedTextField()
    tagline     = LocalizedTextField()
    description = LocalizedTextField()
    methodology_note = LocalizedTextField()
    # ... non-localised: slug, country, registration_id, etc.
```

### API serialisation

A reusable `LocalizedSerializerField` emits the nested shape on the wire:

```python
# apps/core/serializers.py
class LocalizedSerializerField(serializers.JSONField):
    """Serialises {en, ru} dict; ensures both keys present even when empty."""
    def to_representation(self, value):
        v = value or {}
        return {'en': v.get('en', ''), 'ru': v.get('ru', '')}
```

Wire shape (matches `LocalizedString` schema in API_SPEC.md):
```json
{
  "name": {"en": "GiveDirectly", "ru": "GiveDirectly"},
  "tagline": {"en": "Cash transfers to extreme poverty",
              "ru": "Денежные переводы людям в условиях крайней бедности"}
}
```

### Fallback behaviour

Frontend (i18next + DESIGN.md): if user's selected language has empty string, fall back to the other language and render with `<span lang="en">…</span>` for screen-reader correctness (DESIGN.md §9.6). Fallback policy is **client-side only** — the API always returns both keys, even if one is empty. Empty string (not `null`) when a translation is missing.

### Curation workflow

| Field | English source | Russian source |
|---|---|---|
| `Charity.name` | ProPublica/Every.org/CharityBase verbatim | Same as EN unless org has a localised name (most don't); Минюст-sourced records use Cyrillic name |
| `Charity.tagline` | Every.org `description` first sentence; manual fallback | Manual RU translation, only for first ~100 charities at launch |
| `Charity.description` | Every.org full description / source data | Manual RU translation, only for first ~100 charities at launch |
| `Charity.methodology_note` | Templated EN: "This charity is verified because…" | Templated RU: "Организация подтверждена…" (Alex curates 1 template per verification_status) |
| `Cause.name` | Every.org native | RU translation from a static `causes_ru.yaml` overlay maintained by Alex |
| `SourceDocument.label` | Templated by `kind` (e.g. "IRS Form 990 (2024)") | Templated by `kind` (e.g. "Налоговая форма IRS 990 (2024)") |
| `MoneyBreakdown.lines[].label` | Templated EN: "Programs" / "Administration" / "Fundraising" | Templated RU: "Программы" / "Администрация" / "Привлечение средств" |
| `TrustBadge.label` | Issuer name verbatim or curated | Manual RU translation per badge slug |

A small **`TranslationOverlay` table** holds curated RU strings keyed by `(model, instance_id, field)`. Django admin form for `Charity` shows EN value (read-only) + RU input next to each LocalizedTextField. Saving the overlay updates the JSONB.

For frequently-templated values (`SourceDocument.label`, `MoneyBreakdown.lines[].label`, `methodology_note`), the templates live in `apps/i18n/templates/{en,ru}.yaml` — applied at serialisation time, not stored per row.

## Consequences

### Positive
- **Adding a third language is a no-op** in the schema — just start populating `{en, ru, es}` in JSONB. No `ALTER TABLE` chain.
- **API shape matches DESIGN.md §13 exactly** — designer-preferred nested form.
- **Bilingual full-text search "for free"**: tsvector indexes both `name->>'en'` AND `name->>'ru'` (ADR-005).
- **No magic** — `JSONField` is stock Django; no `django-modeltranslation` runtime trickery.
- **Operationally simple**: one Charity row, one source of truth, no `Charity_RU`-shaped duplication.

### Negative
- **JSONB queries are syntactically ugly**: filtering on `name->>'en'` is verbose. Mitigated by `LocalizedTextField.lookups` overrides if needed.
- **Schema validators** don't enforce both-keys-present at the DB level — relies on field default + serializer enforcement. Acceptable trade.
- **Manual translation work**: ~30 charities × ~3 fields = ~90 strings to curate at launch. Alex's commitment per SPEC §3 row 6 — captured as Phase 4 task.
- **Search vector trigger** is more complex (concatenates EN + RU via JSONB key access) — encoded in ADR-005.

### Neutral
- Migrating from `LocalizedTextField` to flat columns (or vice versa) later is a one-time data migration script, not a live-traffic concern.

## Alternatives considered

| Alternative | Rejected because |
|---|---|
| **Separate `name_en` / `name_ru` columns** | Adding a third language = ALTER TABLE for every localised field across every model. Brittle. |
| **`django-modeltranslation`** | Adds runtime metaclass magic; documents the *opposite* of what's stored (transparent fields with hidden `_en`/`_ru` suffix columns). Hard to reason about + hard to debug. |
| **`django-parler`** | Translates by maintaining a sibling `CharityTranslation` table joined per language. Heavier (extra JOIN per query) + still needs schema changes for new languages. |
| **`django-rosetta` / `.po` files** | Designed for UI strings, not data content. Wrong tool. |
| **Auto-translate everything via DeepL/Google API** | (a) Cost; (b) charity names should NOT be translated ("GiveDirectly" stays "GiveDirectly"); (c) low-quality auto-translation undermines trust positioning. |
| **Single language at launch, RU added later** | Violates SPEC §3 row 6 (must-have). |
| **Two separate Charity rows for EN and RU** | Breaks dedup (ADR-004), breaks search ranking, doubles row count. |

## Implementation hooks (Phase 3)

- `apps/core/fields.py` — `LocalizedTextField`
- `apps/core/serializers.py` — `LocalizedSerializerField`
- `apps/i18n/management/commands/load_overlay.py` — bulk-load RU translations from a YAML file
- Django admin: registered with custom form per model that renders `LocalizedTextField` as a two-input pair
- Test fixture: `tests/fixtures/charities_bilingual.json` with at least one record having both EN+RU populated, one with EN-only
