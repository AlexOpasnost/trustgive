> **Created**: 2026-05-05 · **Author**: Backend Developer agent

## 0. Executive summary

TrustGive's public API is a **read-mostly REST surface** built on Django REST Framework with `drf-spectacular` for OpenAPI 3.1 generation. There is **no authentication** in MVP (per ADR-002): all 10 endpoints are anonymous. The API serves three classes of consumer:

1. The **TrustGive React frontend** (catalog, detail, comparison, methodology pages).
2. **Server-side renderers / SEO crawlers** consuming `/api/seo/charities/{slug}/` and `/api/feed.rss`.
3. The **TrustGive frontend's own analytics shim** posting to `/api/events/donation-redirect/` (see ADR-008).

Catalog responses are aggressively cacheable (Cloudflare `s-maxage=3600`, see ADR-007). Schema is published at `/api/schema/` and an interactive Swagger UI at `/api/docs/`.

**Conventions**:
- All paths under `/api/` — no `/v1/` prefix in MVP because **drf-spectacular emits `info.version: "1.0.0"`** in the OpenAPI doc and the SPEC commits us to a single-version MVP. Versioning strategy + URL prefix added when we ship `v2` (locked in ADR-003).
- `snake_case` for all JSON keys (Django/DRF convention).
- Slugs as primary URL identifiers (not numeric IDs) for SEO + portability — see ADR-001 for the unique-slug guarantee.
- Localised fields use the **nested object pattern**: `{ "en": "...", "ru": "..." }` per DESIGN.md §13 + ADR-006.
- All list endpoints paginated, default `page_size=20`, max 100.
- Throttling: `AnonRateThrottle` 60 req/min globally; donation-redirect endpoint capped at 10 req/min per IP.
- Error envelope is uniform across every endpoint (see §5).

## 1. Endpoint inventory

| # | Method | Path | Purpose | Cacheable | Throttle |
|---|---|---|---|---|---|
| 1 | GET | `/api/health/` | Liveness + DB + Sentry probe | No | none |
| 2 | GET | `/api/charities/` | Faceted catalog list | 1h CDN | 60/min |
| 3 | GET | `/api/charities/{slug}/` | Charity detail | 1h CDN | 60/min |
| 4 | GET | `/api/charities/{slug}/source-documents/` | Drawer §6.7 list | 24h CDN | 60/min |
| 5 | GET | `/api/charities/compare/` | Side-by-side comparison (max 3) | 1h CDN | 60/min |
| 6 | GET | `/api/causes/` | EN+RU cause taxonomy tree | 24h CDN | 60/min |
| 7 | GET | `/api/feed.rss` | Newly-added charities RSS | 1h CDN | 60/min |
| 8 | GET | `/api/seo/charities/{slug}/` | SEO landing-page payload | 1h CDN | 60/min |
| 9 | POST | `/api/events/donation-redirect/` | Outbound-click logging | No (POST) | 10/min |
| 10 | GET | `/api/schema/` | Raw OpenAPI 3.1 doc | 1h CDN | none |
| 10b | GET | `/api/docs/` | Swagger UI HTML | 1h CDN | none |

> **Why no `/v1/`?** ADR-003 specifies header-prefixed versioning when needed (`Accept-Version: 1`), and a single-version MVP avoids URL-prefix migrations later. drf-spectacular's `info.version` exposes the version contractually.

## 2. OpenAPI 3.1 specification (drf-spectacular compatible)

## 4. Caching + rate-limiting

See ADR-007 for full rationale. Headers per endpoint summarised in §1.

**`Vary` headers** on all cacheable endpoints: `Vary: Accept-Encoding, Accept-Language` so the lang query parameter does not poison the cache when a future header-driven content-negotiation lands.

**Throttle classes** (DRF):
- Global default: `AnonRateThrottle` 60/min
- `/api/events/donation-redirect/`: custom `DonationRedirectThrottle` 10/min keyed on `REMOTE_ADDR`
- `/api/health/`: throttle exempt (so Railway healthcheck never trips it)

**429 response body** matches `ErrorResponse` with `code: RATE_LIMITED` and `details: {retry_after_seconds: int}`.

## 5. Error envelope contract

All errors share the same envelope (per `api-design` skill custom_exception_handler):

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "details": {"field": "slug", "issue": "Invalid slug format"},
    "request_id": "8b1f9d6a-3c4d-4e5f-9a8b-7c6d5e4f3a2b"
  }
}
```

**Standard error codes** (UPPER_SNAKE_CASE, derived from Django/DRF exception class names):
| Code | HTTP | When |
|---|---|---|
| `VALIDATION_ERROR` | 400 | Query/body fails serializer validation |
| `NOT_FOUND` | 404 | Slug or related lookup miss |
| `RATE_LIMITED` | 429 | Throttle tripped — `details.retry_after_seconds` populated |
| `SERVICE_UNAVAILABLE` | 503 | DB unreachable (only `/api/health/` returns this in normal ops) |
| `INTERNAL_ERROR` | 500 | Caught exception; Sentry event id mirrored to `details.sentry_id` for support |

`request_id` is set by middleware (UUID4 per request) and surfaced in `X-Request-ID` response header for log correlation (see ADR-008).

## 6. URL identifier policy: slugs over IDs

Per ADR-001, `slug` is the URL primary key for `Charity`. Slug derivation:

```
slug = slugify(name_en)                                  e.g. "givedirectly"
if collision: slug = f"{slug}-{registration_id_clean}"   e.g. "givedirectly-271661997"
```

`registration_id_clean` strips non-alphanumerics. UUID `id` exists internally for foreign keys but is NOT exposed in URLs.

Trade-off: charity name changes break slug stability. **Mitigation**: maintain a `CharitySlugAlias(charity_id, slug, retired_at)` table. Old slugs `301`-redirect via Django middleware. Alex curates renames manually for v1 (~rare).

## 7. Pagination policy

`PageNumberPagination` (DRF default), `PAGE_SIZE=20`, `max_page_size=100`. Cursor pagination considered and rejected for MVP because:
- Catalog rarely exceeds 50 pages even with 20 results × 1.2M records — most filters narrow heavily
- Page numbers are SEO-friendly (canonical pagination URLs)
- Frontend pagination UI (DESIGN.md §7.2) shows numbered pages — needs `count` total

If a single facet ever returns >10K records and pagination becomes slow (`COUNT(*) OVER()` on large tables), revisit with cursor pagination as a non-breaking opt-in (`?cursor=...` parameter).

## 8. Open questions (none block approval)

1. **Should the SEO endpoint return rendered HTML instead of JSON?** Current shape returns JSON for Frontend SSR to render. If we go static-export route, we may want a `/seo/charities/{slug}.html` variant. **Defer**: revisit when Frontend Developer commits to SSR strategy in Phase 4.
2. **`/api/events/donation-redirect/` — 202 vs 204?** Chose 202 because we want to confirm the event was queued. 204 (no body) would also be valid. **No user impact** either way.
3. **Should `/api/charities/compare/` accept a POST with the slug list in the body** to avoid URL-length issues? URLs cap at ~2KB on most CDNs and 3 slugs are tiny. **Stay GET** for cacheability.

## 10. Known internal inconsistency to resolve in Phase 3

ADR-005 (search) shows a Postgres trigger using `NEW.name_en`/`NEW.name_ru` direct columns, while ADR-006 (i18n) commits us to a single JSONB `name` column via `LocalizedTextField`. Phase 3 implementation must reconcile by rewriting the trigger to read from JSONB:

```sql
NEW.search_vector :=
  setweight(to_tsvector('simple', unaccent(coalesce(NEW.name->>'en', ''))), 'A') ||
  setweight(to_tsvector('simple', unaccent(coalesce(NEW.name->>'ru', ''))), 'A') ||
  setweight(to_tsvector('simple', unaccent(coalesce(NEW.description->>'en', ''))), 'C') ||
  setweight(to_tsvector('simple', unaccent(coalesce(NEW.description->>'ru', ''))), 'C') ||
  setweight(to_tsvector('simple', coalesce(NEW.registration_id, '')), 'B');
```

ADR-006 wins (JSONB is the storage decision). ADR-005's spirit (FTS strategy + trigger-driven population) is preserved.
